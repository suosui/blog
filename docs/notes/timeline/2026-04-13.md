# BridataV5 CI/CD 从零搭建完整手册

适用场景：新服务器 + GitLab 自托管 + 多服务 TypeScript monorepo + PM2 部署

---

## 前置条件

- 一台全新 Ubuntu 服务器（已有公网 IP）
- GitLab 实例已运行（本文示例：`http://gitlab.test.com:8090`）
- 代码已推送到 GitLab，monorepo 结构：
  - `bridata_api_core/`：共享框架库，其他服务通过 git URL 依赖它
  - `comp/ core/ custom/ mod/ resource/ calculator/ log/ proxy/ share/`：9 个业务服务

---

## 第一步：服务器基础环境

```bash
# 更新系统
apt-get update && apt-get upgrade -y

# 安装基础工具
apt-get install -y curl git

# 安装 Node.js 22
curl -fsSL https://deb.nodesource.com/setup_22.x | bash -
apt-get install -y nodejs

# 验证
node -v   # 应输出 v22.x.x
npm -v

# 全局安装 PM2
npm install -g pm2
```

---

## 第二步：创建 deployer 用户

PM2 进程统一由 `deployer` 用户管理，CI Runner 以 root 运行，通过 `su - deployer -c` 委托 PM2 操作。

```bash
# 创建用户
useradd -m -s /bin/bash deployer

# 设置密码（供 su 切换）
passwd deployer

# 为 deployer 安装 PM2（deployer 自己的环境）
su - deployer -c "npm install -g pm2"

# 设置 deployer 环境变量（服务以 test 配置启动）
echo 'export NODE_ENV=test' >> /home/deployer/.bashrc

# 创建日志目录
mkdir -p /var/log/bridata
chown deployer:deployer /var/log/bridata
```

---

## 第三步：创建服务部署目录并拉取代码

```bash
mkdir -p /opt/bridata
cd /opt/bridata

# 每个服务拉取 test 分支
for SERVICE in comp core custom mod resource calculator log proxy share; do
  git clone -b test http://gitlab.test.com:8090/root/${SERVICE}.git ${SERVICE}
done

# 授权给 deployer
chown -R deployer:deployer /opt/bridata
chmod -R 755 /opt/bridata
```

## 第四步：安装 GitLab Runner

```bash
# 下载安装
curl -L "https://packages.gitlab.com/install/repositories/runner/gitlab-runner/script.deb.sh" | bash
apt-get install -y gitlab-runner

# 必须以 root 身份安装（避免 prepare environment 错误）
gitlab-runner install --user root --working-directory /root
gitlab-runner start

# 验证运行状态
systemctl status gitlab-runner
```

---

## 第五步：注册 Runner 到 GitLab

**在 GitLab 获取注册 token**：
进入 `http://gitlab.test.com:8090/admin/runners` → 点击 `New instance runner` → 复制 token

```bash
gitlab-runner register \
  --non-interactive \
  --url "http://gitlab.test.com:8090/" \
  --registration-token "<从GitLab复制的token>" \
  --executor "shell" \
  --description "bridata-deploy-runner" \
  --tag-list "bridata-deploy" \
  --run-untagged="false" \
  --locked="false"

# 验证注册成功
cat /etc/gitlab-runner/config.toml
```

注册完成后，在 GitLab `Admin > Runners` 页面可以看到 runner 变绿。

> **注意**：如果 runner 显示灰色 offline，执行 `gitlab-runner verify` 然后 `systemctl restart gitlab-runner`

## 第六步：安装并配置 Kafka（KRaft 模式，无需 ZooKeeper）

```bash
# 安装 Java（Kafka 依赖）
apt-get install -y default-jdk

# 下载 Kafka（使用清华镜像）
cd /opt
wget https://mirrors.tuna.tsinghua.edu.cn/apache/kafka/3.7.0/kafka_2.13-3.7.0.tgz
tar -xzf kafka_2.13-3.7.0.tgz
mv kafka_2.13-3.7.0 kafka

# KRaft 模式：初始化存储（只需执行一次）
KAFKA_CLUSTER_ID=$(/opt/kafka/bin/kafka-storage.sh random-uuid)
/opt/kafka/bin/kafka-storage.sh format \
  -t $KAFKA_CLUSTER_ID \
  -c /opt/kafka/config/kraft/server.properties

# 创建 systemd 服务（开机自启）
cat > /etc/systemd/system/kafka.service << 'EOF'
[Unit]
Description=Apache Kafka (KRaft)
After=network.target

[Service]
Type=simple
User=root
ExecStart=/opt/kafka/bin/kafka-server-start.sh /opt/kafka/config/kraft/server.properties
ExecStop=/opt/kafka/bin/kafka-server-stop.sh
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target
EOF

systemctl daemon-reload
systemctl enable kafka
systemctl start kafka

# 验证（等 5 秒启动完成）
sleep 5 && nc -z localhost 9092 && echo "Kafka 运行正常"
```

## 第七步：GitLab 配置 — 创建 Personal Access Token

用于 CI 中 npm 拉取 `bridata_api_core` 时的 Git 认证（CI_JOB_TOKEN 只能访问当前项目）。

1. 打开 `http://gitlab.test.com:8090/-/user_settings/personal_access_tokens`
2. 创建 token：
   - Name：`ci-read-token`
   - Scopes：勾选 `read_repository`
3. 复制生成的 token 值（只显示一次）

## 第八步：GitLab 配置 — 设置 CI 变量

### 8.1 在 bridata_api_core 仓库设置变量

进入 `http://gitlab.test.com:8090/root/bridata_api_core/-/settings/ci_cd` → Variables

| 变量名 | 值 | Masked | 说明 |
|---|---|---|---|
| `GITLAB_READ_TOKEN` | 第七步的 PAT | 是 | npm 拉取 bridata_api_core 的认证 |
| `CI_PUSH_TOKEN` | 有 push 权限的 PAT | 是 | build 后 push dist 到 build 分支 |
| `GITLAB_TRIGGER_TOKEN_comp` | comp 的 trigger token | 是 | 见 8.2 |
| `GITLAB_TRIGGER_TOKEN_core` | core 的 trigger token | 是 | |
| `GITLAB_TRIGGER_TOKEN_custom` | custom 的 trigger token | 是 | |
| `GITLAB_TRIGGER_TOKEN_mod` | mod 的 trigger token | 是 | |
| `GITLAB_TRIGGER_TOKEN_resource` | resource 的 trigger token | 是 | |
| `GITLAB_TRIGGER_TOKEN_calculator` | calculator 的 trigger token | 是 | |
| `GITLAB_TRIGGER_TOKEN_log` | log 的 trigger token | 是 | |
| `GITLAB_TRIGGER_TOKEN_proxy` | proxy 的 trigger token | 是 | |
| `GITLAB_TRIGGER_TOKEN_share` | share 的 trigger token | 是 | |

### 8.2 为每个服务创建 Pipeline Trigger Token

对每个服务（comp/core/custom/...）：
1. 进入 `http://gitlab.test.com:8090/root/{service}/-/settings/ci_cd` → Pipeline triggers
2. 点击 Add new token → 复制 token
3. 填入 8.1 的表格对应变量

### 8.3 在各服务仓库也设置 GITLAB_READ_TOKEN

每个服务的 install 阶段也需要这个变量（npm 拉取 bridata_api_core）。

最省事的做法：在 GitLab Group 级别设置一次，所有仓库自动继承：
`http://gitlab.test.com:8090/groups/root/-/settings/ci_cd` → Variables → 添加 `GITLAB_READ_TOKEN`

## 第十步：bridata_api_core 的 CI 配置

文件路径：`bridata_api_core/.gitlab-ci.yml`

```yaml
stages:
  - install
  - build
  - trigger-services

install:
  stage: install
  image: node:22
  script:
    - |
      if [ -d node_modules ] && [ "$(md5sum package-lock.json | awk '{print $1}')" = "$(cat node_modules/.package-lock.md5 2>/dev/null)" ]; then
        echo "依赖未变化，跳过 npm install"
      else
        npm install --registry=https://registry.npmmirror.com
        md5sum package-lock.json | awk '{print $1}' > node_modules/.package-lock.md5
      fi
  cache:
    key: npm-cache
    paths:
      - node_modules/
  only:
    - build

build:
  stage: build
  image: node:22
  script:
    - rm -rf dist
    - npm run build
    - git config --global user.email "ci@bridata.com"
    - git config --global user.name "CI Bot"
    - git add dist -f
    - git commit -m "chore(ci): auto build dist [skip ci]" || echo "No changes to commit"
    - git push http://oauth2:${CI_PUSH_TOKEN}@gitlab.test.com:8090/root/bridata_api_core.git HEAD:build
  cache:
    key: npm-cache
    paths:
      - node_modules/
  artifacts:
    paths:
      - dist/
    expire_in: 1 week
  tags:
    - bridata-api-core-new
  only:
    - build

trigger-services:
  stage: trigger-services
  tags:
    - bridata-api-core-new
  script:
    - |
      GITLAB_URL="http://gitlab.test.com:8090"

      declare -A SERVICE_IDS=(
        ["comp"]="228"
        ["core"]="219"
        ["custom"]="220"
        ["mod"]="217"
        ["resource"]="229"
        ["calculator"]="231"
        ["log"]="218"
        ["proxy"]="227"
        ["share"]="230"
      )

      declare -A TRIGGER_TOKENS=(
        ["comp"]="${GITLAB_TRIGGER_TOKEN_comp}"
        ["core"]="${GITLAB_TRIGGER_TOKEN_core}"
        ["custom"]="${GITLAB_TRIGGER_TOKEN_custom}"
        ["mod"]="${GITLAB_TRIGGER_TOKEN_mod}"
        ["resource"]="${GITLAB_TRIGGER_TOKEN_resource}"
        ["calculator"]="${GITLAB_TRIGGER_TOKEN_calculator}"
        ["log"]="${GITLAB_TRIGGER_TOKEN_log}"
        ["proxy"]="${GITLAB_TRIGGER_TOKEN_proxy}"
        ["share"]="${GITLAB_TRIGGER_TOKEN_share}"
      )

      FAILED=""
      for SERVICE in "${!SERVICE_IDS[@]}"; do
        PROJECT_ID="${SERVICE_IDS[$SERVICE]}"
        TOKEN="${TRIGGER_TOKENS[$SERVICE]}"
        echo "触发服务 $SERVICE 的 pipeline（Project ID: $PROJECT_ID）..."
        HTTP_STATUS=$(curl -s -o /tmp/curl_out -w "%{http_code}" -X POST \
          "${GITLAB_URL}/api/v4/projects/${PROJECT_ID}/trigger/pipeline" \
          --form "token=${TOKEN}" \
          --form "ref=test" \
          --form "variables[TRIGGERED_BY]=bridata_api_core" \
          --form "variables[CORE_BUILD_SHA]=${CI_COMMIT_SHA}")
        if [ "$HTTP_STATUS" -ge 200 ] && [ "$HTTP_STATUS" -lt 300 ]; then
          echo "已触发: $SERVICE (HTTP $HTTP_STATUS)"
        else
          echo "触发失败: $SERVICE (HTTP $HTTP_STATUS)"
          cat /tmp/curl_out
          FAILED="$FAILED $SERVICE"
        fi
      done
      if [ -n "$FAILED" ]; then
        echo "以下服务触发失败:$FAILED"
        exit 1
      fi
  only:
    - build
```

## 第十一步：各业务服务的 CI 配置（以 core 为例）

文件路径：`{service}/.gitlab-ci.yml`，每个服务只有 `SERVICE_NAME`、`PM2_PROCESS_NAME`、`APP_DIR` 三个变量不同。

```yaml
variables:
  SERVICE_NAME: "core"                    # 改为对应服务名
  PM2_PROCESS_NAME: "bridata-core"        # 改为对应 PM2 进程名
  APP_DIR: "/opt/bridata/core"            # 改为对应目录

stages:
  - install
  - build
  - deploy

install:
  stage: install
  tags:
    - bridata-deploy
  script:
    - cd ${APP_DIR}
    - |
      if [ ! -d ".git" ]; then
        git init
      fi
      git remote set-url origin "${CI_REPOSITORY_URL}" 2>/dev/null || git remote add origin "${CI_REPOSITORY_URL}"
      git fetch origin
      git reset --hard origin/${CI_COMMIT_REF_NAME}
    - |
      LOCK_MD5=$(md5sum package-lock.json | awk '{print $1}')
      CACHED_MD5=$(cat .install-cache-md5 2>/dev/null || echo "none")
      git config --global url."http://oauth2:${GITLAB_READ_TOKEN}@gitlab.test.com:8090/".insteadOf "http://gitlab.test.com:8090/"
      if [ "$TRIGGERED_BY" = "bridata_api_core" ]; then
        echo "更新 bridata_api_core（由 core 触发）..."
        npm update bridata_api_core --registry=https://registry.npmmirror.com
        LOCK_MD5=$(md5sum package-lock.json | awk '{print $1}')
        echo "$LOCK_MD5" > .install-cache-md5
      elif [ "$LOCK_MD5" != "$CACHED_MD5" ]; then
        echo "执行 npm install（lock 变化）..."
        npm install --registry=https://registry.npmmirror.com
        echo "$LOCK_MD5" > .install-cache-md5
      else
        echo "跳过 npm install（依赖无变化）"
      fi
      git config --global --unset url."http://oauth2:${GITLAB_READ_TOKEN}@gitlab.test.com:8090/".insteadOf
  only:
    - test

build:
  stage: build
  tags:
    - bridata-deploy
  script:
    - cd ${APP_DIR}
    - npm run build
  only:
    - test

deploy:
  stage: deploy
  tags:
    - bridata-deploy
  script:
    - |
      if su - deployer -c "pm2 describe ${PM2_PROCESS_NAME}" > /dev/null 2>&1; then
        echo "重载进程: ${PM2_PROCESS_NAME}"
        su - deployer -c "pm2 reload ${PM2_PROCESS_NAME} --update-env"
      else
        echo "首次启动进程: ${PM2_PROCESS_NAME}"
        mkdir -p /var/log/bridata
        su - deployer -c "NODE_ENV=test pm2 start ${APP_DIR}/dist/bin.js \
          --name ${PM2_PROCESS_NAME} \
          --cwd ${APP_DIR} \
          --log /var/log/bridata/${SERVICE_NAME}.log \
          --error /var/log/bridata/${SERVICE_NAME}-error.log \
          --time \
          --restart-delay 3000 \
          --max-restarts 5"
      fi
    - su - deployer -c "pm2 save"
    - |
      echo "健康检查中..."
      for i in $(seq 1 10); do
        STATUS=$(su - deployer -c "pm2 jlist" | node -e "
          const list = JSON.parse(require('fs').readFileSync('/dev/stdin','utf8'));
          const p = list.find(p => p.name === '${PM2_PROCESS_NAME}');
          console.log(p ? p.pm2_env.status : 'not_found');
        ")
        echo "  状态: $STATUS (第 ${i}/10 次检查)"
        [ "$STATUS" = "online" ] && { echo "部署成功: ${PM2_PROCESS_NAME}"; exit 0; }
        [ $i -eq 10 ] && { echo "部署失败，输出最近日志:"; su - deployer -c "pm2 logs ${PM2_PROCESS_NAME} --lines 30 --nostream"; exit 1; }
        sleep 3
      done
  only:
    - test
```

|---|---|---|---|

| core | core | bridata-core | /opt/bridata/core |
| custom | custom | bridata-custom | /opt/bridata/custom |
| mod | mod | bridata-mod | /opt/bridata/mod |
| resource | resource | bridata-resource | /opt/bridata/resource |
| calculator | calculator | bridata-calculator | /opt/bridata/calculator |

## 常见问题速查

| 现象 | 原因 | 修复 |
|---|---|---|
| Runner 灰色 offline | Runner 以 deployer 注册，restart 无权限 | 以 root 重新注册，`/etc/gitlab-runner/config.toml` |
| `prepare environment: exit status 1` | Runner install 时 `--user deployer` | `gitlab-runner install --user root` |
| `Host key verification failed` | git remote 是 SSH，root 无 SSH Key | CI 脚本中用 `git remote set-url origin "${CI_REPOSITORY_URL}"` |
| `HTTP Basic: Access denied`（npm install）| CI_JOB_TOKEN 无法跨项目 | 创建 PAT，设置 `git config --global url.insteadOf` |
| bridata_api_core 版本未更新 | package-lock.json 锁定旧 commit SHA | 级联触发时用 `npm update bridata_api_core` 而非 `npm install` |
| 某服务 trigger 400，后续服务未触发 | curl `--fail` 直接退出 | 改用 `curl -w "%{http_code}"` 捕获状态码，遍历完再 exit |
| Kafka `No readable meta.properties` | KRaft 模式未初始化 | `kafka-storage.sh format -t $(kafka-storage.sh random-uuid) -c server.properties` |
| PM2 列表为空 | deploy 阶段未运行到（build 先失败了）| 先保证 build 通过，deploy 会自动 `pm2 start` |
