### 同步/异步 IO 和 阻塞/非阻塞 IO 的区别

这两组概念描述的是不同维度的问题：

- **阻塞/非阻塞**：关注的是调用者在等待结果时是否被挂起（发起 IO 调用后，线程是否可以立刻返回去做别的事）
- **同步/异步**：关注的是 IO 操作完成后，是谁负责把数据搬到用户空间、并通知调用者（数据准备好之后的处理方式）

#### 阻塞 vs 非阻塞

以 `read()` 为例：

| | 行为 |
|---|---|
| **阻塞 IO** | 调用 `read()` 后，线程挂起，直到数据准备好并拷贝完成才返回 |
| **非阻塞 IO** | 调用 `read()` 后立刻返回（数据没准备好就返回 `EWOULDBLOCK`），调用者需要不断轮询（poll）检查数据是否就绪 |

#### 同步 vs 异步

| | 行为 |
|---|---|
| **同步 IO** | 调用者需要自己去"等待/检查"数据是否就绪，并且由调用者线程亲自完成数据从内核态到用户态的拷贝 |
| **异步 IO** | 调用者发起请求后完全不用管，内核负责等待数据就绪 + 拷贝数据，完成后通过回调/信号/事件通知调用者，调用者拿到的就是现成的结果 |

#### 一张图理清四种组合

一次 `read()` 网络数据的过程分两步：

1. 等待数据从网络到达内核缓冲区（可能耗时）
2. 把数据从内核缓冲区拷贝到用户缓冲区（通常很快）

| 模型 | 第①阶段（等待就绪） | 第②阶段（拷贝数据） |
|---|---|---|
| 阻塞 IO | 线程挂起等待 | 线程亲自拷贝（仍阻塞） |
| 非阻塞 IO | 立即返回，需轮询 | 数据就绪后，线程亲自拷贝（这一步仍阻塞，但很短） |
| IO 多路复用（select/epoll） | 通过 select/epoll 阻塞等待多个 fd | 数据就绪后，线程亲自拷贝 |
| **异步 IO（AIO）** | 内核负责等待 | **内核负责拷贝**，完成后通知 |

#### 关键结论

- **阻塞/非阻塞** 是同步 IO 内部的两种子模式（Unix Network Programming 中的经典分类），它们本质上都是同步 IO——因为最终的数据拷贝都是调用者线程去做的，只是"等"的方式不同（挂起 vs 轮询）。
- 只有真正的异步 IO（如 Linux 的 `io_uring`、Windows 的 IOCP）才把"等待就绪"和"数据拷贝"两步都交给内核/系统完成，调用者只需要注册回调，等通知即可。
- 常见误区：把 `epoll`/`select` + 非阻塞 socket 称为"异步 IO"其实不准确，严格来说这仍是同步的 IO 多路复用模型（Node.js、Nginx 的事件循环本质上都是这一类），因为拷贝数据那一步依然是用户线程主动完成的。

一句话总结：**阻塞/非阻塞决定了"发起调用后你是否马上能干别的事"；同步/异步决定了"数据到位后是谁把活干完的"**。
