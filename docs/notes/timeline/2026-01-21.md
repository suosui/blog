# 多 Agent 协作的员工绩效管理系统研发方案

## 一、项目目标

构建一个 **员工绩效管理系统**，通过多 Agent（多角色）协作的方式，模拟真实研发流程，实现从需求 → 设计 → 开发 → 测试 → 交付的完整闭环。

核心理念：**文件即协作、文件即契约、文件即成果**。

---

## 二、技术栈

* **前端**：Vue3
* **后端**：Node.js + TypeScript
* **数据库**：MySQL
* **接口回归测试**：Postman + Newman
* **前端 E2E 测试**：Playwright 或 Cypress

---

## 三、协作模式概述

### 3.1 单一项目目录 + 多终端

在同一个新建的项目文件夹下，打开 **6 个 Terminal（或 Tab）**，每个 Terminal 对应一个独立角色 Agent。

### 3.2 角色与终端映射

| Terminal | 角色             | 说明                 | 使用工具   |
| -------- | -------------- | ------------------ | ------ |
| 1        | 项目经理（PM Agent） | 唯一指挥者，负责任务拆解与验收    | Codex  |
| 2        | 产品经理 Agent     | 输出 PRD / 用户故事 / AC | Codex  |
| 3        | UI 设计师 Agent   | 输出原型、设计稿、设计规范      | Codex  |
| 4        | 前端工程师 Agent    | 实现 Vue3 前端         | Claude |
| 5        | 后端工程师 Agent    | 实现 Node.js + TS 后端 | Claude |
| 6        | 测试工程师 Agent    | 编写并执行自动化测试         | Gemini |

---

## 四、协作与指挥原则

1. **唯一指挥者原则**

   * 只有 PM Agent 具备指挥权
   * 你只向 PM Agent 下发需求

2. **文件驱动原则**

   * 每一步必须产出文件
   * PM 只认文件，不认口头描述
   * 所有产出必须落地到项目目录

3. **可审计原则**

   * 所有阶段成果可版本化（Git）
   * 任一阶段可回溯、可复现

## 六、标准流程（文件交付闭环）

### Step 1：项目经理（PM）

**输入**：你的原始需求描述

**产出文件**：

* `docs/pm/milestones.md`（里程碑与阶段目标）
* `docs/pm/task-list.md`（任务拆解、责任人、交付物）

---

### Step 2：产品经理 Agent

**PM 指派产品经理 Agent**

**产出文件**：

* `docs/product/prd.md`
* `docs/product/user-stories.md`
* `docs/product/acceptance-criteria.md`

> 验收标准（AC）必须可被自动化测试执行

---

### Step 3：UI 设计师 Agent

**PM 基于 PRD 指派 UI Agent**

**产出文件**：

* `docs/ui/information-architecture.md`
* `docs/ui/design-system.md`
* `docs/ui/wireframes/`
* `docs/ui/screens/`

---

### Step 4：后端工程师 Agent

**基于 PRD + 原型实现后端**

**产出文件**：

* `docs/api/openapi.yaml`（契约优先）
* `docs/api/db-schema.sql`
* `docs/api/data-dict.md`
* `backend/` 后端实现代码

---

### Step 5：前端工程师 Agent

**基于 OpenAPI + UI 规范实现前端**

**产出文件**：

* `frontend/` Vue3 项目代码
* （可选）`docs/frontend/pages.md`

---

### Step 6：测试工程师 Agent

**将 AC 转为可执行测试并运行**

**产出文件**：

* `tests/api/collection.json`
* `tests/api/env.json`
* `tests/api/newman.sh`
* `tests/e2e/` 自动化测试代码

**测试结果规则**：

* 通过：通知 PM + 提供测试报告路径
* 不通过：同时通知 PM、前端 Agent、后端 Agent，并附失败用例与日志

---

### Step 7：PM 闭环管理

* 汇总失败项 → `docs/pm/bugs.md`
* 更新任务 → `docs/pm/task-list.md`
* 指派修复 → 重新测试 → 直到 CI 全绿

---

## 七、Agent 配置文件落地原则

* **Codex Agent**：`.codex/`
* **Claude Agent**：`.claude/agents/`
* **Gemini Agent**：`.gemini/`

所有 Agent 的 Prompt、规则、工作协议必须：

* 以文件形式存在
* 可复用、可版本化
* 不依赖手动复制粘贴到 Terminal

---

## 八、完成判定标准（PM 视角）

* 所有阶段均有明确文件产出
* OpenAPI 与前后端实现一致
* Postman/Newman 回归通过
* 前端 E2E 测试通过
* CI 全绿

> **只有 CI 全绿，PM 才判定项目完成**
