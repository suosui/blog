# Vibe Coding
## 从「写代码」到「驾驭代码」的新开发范式

分享人：赵成帮  
日期：2026-01-23

---

## 一个问题

你最近写代码时，有没有这种感觉？

- 大量 CRUD / 样板代码  
- 时间花在拼接口、查文档  
- 真正思考架构的时间越来越少  

**问题是：**
> 我们真的还需要亲手写完每一行代码吗？

---

## 什么是 Vibe Coding？

**一句话定义：**

> Vibe Coding = 人负责意图，AI 负责实现

- 人：目标、约束、判断
- AI：生成代码、补全、重构、测试

---

## Vibe Coding ≠ 不会写代码

这是一个常见误解：

- ❌ 不会写代码
- ❌ 逃避工程复杂度

而是：

> **不值得亲手写的代码，交给 AI**

工程师价值正在转变为：
- 架构能力
- 判断能力
- 系统理解能力

---

## 为什么会出现 Vibe Coding？

三个背景同时发生：

1. **大模型代码能力质变**
2. **工程复杂度指数级上升**
3. **人类时间成为瓶颈**

结论：

> 写代码不再是稀缺能力  
> 做出正确决策才是

---

## 传统 Coding vs Vibe Coding

| 对比项 | 传统 Coding | Vibe Coding |
|------|------------|------------|
| 写代码 | 人 | AI |
| 思考重点 | 语法 / API | 架构 / 边界 |
| 生产力 | 线性 | 放大 |
| 工程师角色 | 实现者 | 指挥官 |

---

## 核心转变：从 How 到 What

传统关注：
- 怎么写
- 用哪个 API
- 参数怎么传

Vibe Coding 关注：
- 我要什么结构
- 有什么约束
- 怎么验证是对的

---

## 核心原则一：人负责决策

AI 不应该做的事：

- 业务取舍
- 架构拍板
- 上线决策

> **所有系统后果，必须由人负责**

---

## 核心原则二：Prompt 是新代码

Prompt ≠ 问问题  
Prompt = **规格说明书**

一个好 Prompt 包含：
- 上下文
- 约束
- 输出要求
