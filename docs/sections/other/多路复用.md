### 同步/异步 IO 和 阻塞/非阻塞 IO 的区别

这两组概念描述的是两个互相独立的维度：

- **阻塞/非阻塞**：关注的是等待方式——发起调用后，调用者是被挂起等待，还是可以立刻返回去做别的事
- **同步/异步**：关注的是通知方式——IO 完成后，是调用者主动去获取结果，还是由操作系统主动通知调用者

> "数据由谁负责拷贝"只是 Linux 网络 IO 模型里同步/异步的一个具体表现，并不是通用定义。磁盘 IO、DMA、其他设备 IO 未必都能简单归纳成"谁搬数据"，经典教材（如 UNIX Network Programming）对同步/异步的定义就是"主动获取结果"与"内核完成后通知你"这一条。

#### 阻塞 vs 非阻塞

以 `read()` 为例：

| | 行为 |
|---|---|
| **阻塞 IO** | 调用 `read()` 后，线程挂起，直到内核准备好数据并完成拷贝才返回 |
| **非阻塞 IO** | 调用 `read()` 后立刻返回（数据没准备好就返回 `EWOULDBLOCK`），调用者需要不断轮询（poll）检查数据是否就绪 |

#### 同步 vs 异步

| | 行为 |
|---|---|
| **同步 IO** | 调用者主动发起数据读取，线程一直等待，直到内核完成数据拷贝（`copy_to_user()`）并返回用户态 |
| **异步 IO** | 调用者发起请求后立刻返回、不用等待，内核负责等待数据就绪并完成拷贝，完成后通过回调/信号/事件通知调用者，调用者直接拿到现成的结果 |

真正执行数据拷贝的始终是内核（CPU 进入内核态执行 `copy_to_user()`），用户线程本身并不会去做 `memcpy`。区别只在于：这份拷贝完成后，是调用者的线程一直阻塞等着，还是调用者压根没等，由内核异步通知。

#### 一张图理清四种组合

一次 `read()` 网络数据的过程分两步：

1. 等待数据从网络到达内核缓冲区（可能耗时）
2. 把数据从内核缓冲区拷贝到用户缓冲区（由内核的 `copy_to_user()` 完成，通常很快）

| 模型 | 第①阶段（等待就绪） | 第②阶段（拷贝数据） |
|---|---|---|
| 阻塞 IO | 线程挂起等待 | 内核拷贝，线程同步等待拷贝完成 |
| 非阻塞 IO | 立即返回，需轮询 | 数据就绪后，内核拷贝，线程同步等待这一小段拷贝完成 |
| IO 多路复用（select/epoll） | 通过 select/epoll 阻塞等待多个 fd | 应用线程调用 `read()` 后，由内核完成拷贝，线程同步等待 `read()` 返回 |
| **异步 IO（AIO）** | 内核负责等待 | **内核负责拷贝**，完成后通知，调用者全程不等待 |

#### 关键结论

- **阻塞/非阻塞** 和 **同步/异步** 是两个独立的维度，不是包含关系。例如 Java 的 `Future.get()` 就是"异步 + 阻塞"：任务本身是异步提交执行的，但调用 `get()` 的线程会阻塞等待结果；反过来同步操作也未必阻塞（可以配合轮询实现非阻塞的同步 IO）。"阻塞"本身也不是 IO 专属概念，`sleep()`、`Condition.await()`、`join()` 同样会阻塞，但和 IO 模型无关。
- 只是在经典的 Linux 网络 IO 模型中，阻塞 IO、非阻塞 IO、IO 多路复用、信号驱动 IO 这四种恰好都属于同步 IO——因为它们都是调用者主动发起 `read()` 并等待内核完成拷贝；真正的异步 IO 只有 Asynchronous I/O（如 Linux AIO、`io_uring`、Windows 的 IOCP）。
- 常见误区：把 `epoll`/`select` + 非阻塞 socket 称为"异步 IO"其实不准确，严格来说这仍是同步的 IO 多路复用模型（Node.js、Nginx 的事件循环本质上都是这一类）——应用线程调用 `read()` 后，虽然由内核完成拷贝，但应用线程是同步等待 `read()` 返回的，并没有真正解耦"等待"和"通知"。
- Linux AIO 本身也是真正的异步 IO，只是它对普通文件的支持不好、API 难用、性能一般，所以现代 Linux 更推荐用 `io_uring` 作为高性能异步 IO 机制。

一句话总结：**阻塞/非阻塞决定了"发起调用后你是否马上能干别的事"；同步/异步决定了"数据到位后是谁负责通知并交付结果"**。
